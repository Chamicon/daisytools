Throughout the section there will be pseudocode examples written to simplify programming, scripting and markup languages.

2. Concept

The validator engine DLL is based upon several validation objects. Each object is dedicated to a specific validation purpose and could be used separate. There is also a number of relation objects that are handling relations in between different validation objects, these objects are not to be used as stand alone objects. Finally there is a couple of communication objects used for communication in between the validation engine and the outside world.

Because of the current structure of the validator engine, there is no possibility to do multiple validations at the same time. There is also no support for validation via http etc. Further, there is no current support for protected DTBs.

3. Dependencies

The validator engine DLL makes use of a number of external components. Some of the components are custom made for the Validator Engine DLL but most are commonly available components found on any normal windows machine.

Dependencies
MSXML 4.0 or higher
Direct X 7 or higher
Visual Basic 6.0 runtime module
DTDParser.dll - custom made DLL

4 Description of the validation objects

The validation objects are to be used as standalone objects for user validation, some of these objects are internally instantiated and called by other validation objects. These objects simple to use and requires no preparations.

The various validation objects:

Validation objects Validation type
oValidateDTB Validation of a whole DTB file set
oValidateMultiVolume Validation of a whole multi-volume DTB file set
oValidateDiscinfo Validation of a single discinfo file
oValidateNcc Validation of a single ncc file
oValidateSmil Validation of a single smil file
oValidateContent Validation of a single content (text source) file
oValidateMasterSmil Validation of a single master smil file

All validation objects has one public function named 'fncValidate' and one public object named 'objReport'. The 'fncValidate' function is usually called with one argument, the absolute path and filename of the target of validation. All errors that are discovered during the validation will be placed in the public 'objReport' object which is of type oReport. After validation the validation object will return a boolean value depending on it's success of validation. This boolean value will only be set to 'false' if an internal processing error occurred, regardless of any failing tests.

Description of the 'fncValidate' arguments for each validation object:

Validation Object Function and arguments Argument usage description
oValidateDTB fncValidate( isAbsPath as string ) as boolean isAbsPath must be the absolute path
oValidateMultiVolume fncValidate( isAbsPath() as string ) as boolean isAbsPath must be all the absolute paths to all the volumes in the multivolume DTB fileset
oValidateDiscinfo fncValidate( isAbsPath as string ) as boolean isAbsPath must be the absolute path and filename to the discinfo file
oValidateNcc fncValidate( isAbsPath as string,
optional ibolNccIsMultivolume as boolean ) as boolean		 isAbsPath must be the absolute path and filename to the ncc file,
ibolNccIsMultiVolume must be set to true if the ncc is part of a multivolume DTB
oValidateSmil fncValidate( isAbsPath as string ) as boolean isAbsPath must be the absolute path and filename to the smil file
oValidateContent fncValidate( isAbsPath as string ) as boolean isAbsPath must be the absolute path and filename to the content file
oValidateMasterSmil fncValidate( isAbsPath as string ) as boolean isAbsPath must be the absolute path and filename to the master smil file

5 Description of the relations objects

The relation objects differs from the validation objects in the way that they are not intended for stand alone validation. Before using one of these objects the user must first validate the data intended for validation trough the corresponding validation object. Usage of these objects is limited since there are already defined validation objects for most validation purposes.

The various relation objects:

Relation objects Relation validation
oNccSmilRelations Validation of the relations in between a NCC file and it's referred SMIL files
oSmilMediaRelations Validation of the relations in between a SMIL files and it's media object files
oMasterSmilRel Validation of the relations in between a master SMIL files and it's referred SMIL files

The relations objects has, like the validation objects, a 'fncValidate' function. The relations objects differs from the validation objects in that way that they don't have a 'objReport' object. The user has to supply the 'fncValidate' function with an already initialized variable of type oReport.

Description of the 'fncValidate' arguments for all relations objects

Function and arguments Argument usage description
fncValidate( iobjReport as oReport, isAbsPath as string, [ibolMultiVolume as boolean] ) as boolean iobjReport must be an initialized object of type oReport, isAbsPath must be the absolute path and filename of the controlling file. ibolMultiVolume is an optional boolean that must be set to 'true' if the object is part of a multi volume.

Description of the 'fncValidate' controlling file ( isAbsPath argument) for each relation object

Relation object Controlling file
oMasterSmilRel Master SMIL file
oNccSmilRelations NCC file
oSmilMediaRelations SMIL file

6 Description of the communication objects

The communication objects handles the communication in between the validator engine and the outside world. There are two types of communication objects; the report objects which contains validation report data and the control objects which handles commands and event messages.

The report objects

The report objects contains report information and provides detailed information about each failed test.

Report object Content
oReport Contains a set of oReportItem objects which together form a whole report
oReportItem Contains information about a single error

The oReportItem object contains information about each separate error. This information includes name, id, short description, long description, fail type, fail class, file, line, column, comment and link. Some of this information is dynamically collected from the VTM.

oReportItem property Description
sTestId As String Contains the VTM test id of the error
sName As String Contains the VTM name of the test
sShortDesc As String Contains the VTM short description of the error
sLongDesc As String Contains the VTM long description of the error
sFailType As String Contains the VTM fail type, is either "error" or "warning"
sFailClass As String Contains the VTM fail class, is either "critical" or "non-critical"
sAbsPath As String Contains the absolute path and filename of the file where the error occured
lColumn As String Contains the column in the file where the error occured
lLine As String Contains the line in the file where the error occured
sComment As String Contains comments and suggestions for change
sLink As String Contains the VTM link to the specification that has info about the error

The oReport object contains a collection of oReportItems, information about how many tests has failed and how many has passed.

oReport public function or property Arguments Description Comment
fncInsertFailedTest ( isTestId as string, [isAbsPath as string], [ilLine as long], [ilColumn as long], [ivComment as variant] ) as boolean Creates a new oReportItem instance in the internal array and fills it with the given information isTestId is a VTM test id, isAbsPath is the absolute path and filename of the failing file (if test is performed on a specific file), ilLine is the line in the file where the error occured, ilColumn is the column in the file where the error occured, isComment is for comments and suggestions of change.
fncRetrieveFailedTestItem ( lItem as long, tItem as oReportItem ) as boolean Retrieves a pointer to a oReportItem in the internal array lItem is a variable containing the number of the report item to retrieve, tItem is an object that will receive a pointer to the specified report item.
subInsertSuccededTest ( ) Increases the lSucceededTestCount variable -
lFailedTestCount as long - Contains the number of error report items within the oReport -
lSucceededTestCount as long - Contains the number of tests succeeded -


Example of how to use the oReport class
/* This is an example of how to insert and retrieve error report items in the oReport object */
declare myReport as new oReport
declare myReportItem as oReportItem
declare lErrorItem as long

/* Notice in the declaration that the 'myReportItem' is not declared as 'new' since it is only going to be used for retrieving pointers to already existing data.

The following lines of code will insert the error 'smilTest.seqDur' and say that the error is located in 'c:\test1.smil' on line 8 column 20. The rest of the information is left out since it's optional. After this we will present the errors to the user by using 'fncRetrieveFailedTestItem' */

myReport.fncInsertFailedTest "smilTest.seqDur", "c:\test1.smil", 8, 20

[..]

for lErrorItem = 0 to myReport.lFailedTestCount - 1
  myReport.fncRetrieveFailedTestitem lErrorItem, myReportItem
  print "error " & myReportItem.name & " found in file " & myReportItem.sAbsPath & "
    at line " & myReportItem.lLine & " column " & myReportItem.lColumn & "."
next lErrorItem

7. The VTM file

The VTM (Validator Test Map) file contains detailed information about each test performed by the validator.

Concept

The validator engine gathers information for each test from the VTM, this is the same information reported in the validator software, since it is built upon the validator engine. Thus altering the VTM changes the actual internal data content of the validator engines reports.

The VTM is an XML document with a custom document type. This document type declares several elements among which the <test> element forms the very core of the VTM. Other elements that are used are <category> and <subcategory> whose purpose is to group and categorize tests.

The VTM is taking advantage of XML to create templates and an advanced inheritance system.

The <test> element

The <test> element contains information for actual tests performed by the validator.

Attribute Note Description
context The test ID
name The test name
shortDesc The short test fail description of the test
longDesc The whole test fail description of the test
failType The fail type of the test, must be 'error' or 'warning'
failClass The fail class of the test. Must be 'critical' or 'non-critical'. Only applies if failType is type 'error'
link Link to a document describing the correct behavior of the tested item. Common use is to link to specifications

The <test> attribute 'context' MUST contain a validator engine ID for the validator engine to be able to use the input data.

Attributes other than 'context' can have any or no data. This data can be altered to suit different languages, quality levels of validation etc.

Test element example
<test context="fileIsReadable" name="File is readable" shortDesc="File is not readable" failType="error" failClass="critical" />

Inheritance

Many tests are used several times for different validation purposes within the validator engine. To be able to differ the tests from each other the VTM uses contexts. A chain of contexts is written as a full string with a '.' separating each context, at the end of each context, a validator engine test id is found.

Context example
The test 'fileIsReadable' is used for validating the readability of each single file type. To separate the different tests, contexts are used.

As an example, while validating a SMIL file the chain of contexts could look something like 'smil.fileIsReadable' as an opposite to the NCC test that would be named 'ncc.fileIsReadable'.

As stated, only the last part of the context chain is an actual test id, the rest is mere contexts. The contexts are declared within the VTM with the elements <category> and <subcategory>.

Element Name Attribute Note
category - Used to group subcategories into a specific context
context The context of the grouped subgroups
name The name of the context group
subcategory - Used to group tests or other subcategories into a specific context
context The context of the grouped tests
name The name of the context subgroup

To place a test within a specific context, the user must place the specific <test> element within a <category> or <subcategory> element.

VTM context declaration example
<subcategory context='SMIL'>
    <test id='fileIsReadable' [..] >
</subcategory>

This method would not suffice for some tests though, therefore the use of more extended contexts is very common within the VTM.

Single DTB validation for example, validates all files within the DTB fileset and would have to separate the different tests with adding another context to the context chain.

Extended context example
dtb.smil.fileIsReadable
dtb.ncc.fileIsReadable

Entities

The frequent repetition of identical tests within different contexts calls for a way to create templates to prevent excessive writing. XML document type declarations offers entity declaration which is used within the VTM as a way of creating templates.

While there is not only single tests that are frequently repeated, but whole set of tests, entities are also used to create templates for whole contexts. While most tests are used more than once, the most efficient way of creating the VTM document is to create all contexts as entities.

Entity examples
<!ENTITY fileExists "
<test context='exists' name='file exists' shortDesc='file not found' longDesc='file not found' failType='error' failClass='critical' link=''/>
" >

<!ENTITY fileIntegrity "
<category context='SMIL'>
&fileExists;
[..]
</category>
" >

Contexts could provide problems while trying to differ from the 'normal' case of behavior. For example, the 'exists' test is testing if a file exists and is used for all file types (NCC/smil/mastersmil/discinfo/content/audio). This test should in almost all cases give a critical error. One example where this test should give a warning instead of a critical error is when the master SMIL file is missing during a full DTB validation.

Overriding this is very simple. Declaring a new element with the same context chain makes the validator engine ignore the previous instance (within the current context only).

Using entities as templates and overriding them
<!-- Declaring the entity for the 'exists' test -->
<!ENTITY fileExists "
< test context='exists' name='file exists' shortDesc='file not found' longDesc='file not found' failType='error' failClass='critical' link=''/>
" >

<!-- The context 'fileIntegrity' integrates the 'fileExists' entity. The 'fileIntegrity' context is used by many functions within the validator engine. -->
<!ENTITY fileIntegrity "
<subcategory name='file validation' context='fi'> &fileExists;
[..]
</subcategory>
" >

<!-- The context 'mastersmil_internal' is just one of the contexts that integrates the 'fileIntegrity' entity. The 'mastersmil_internal' context is used for single master SMIL validation and by the full DTB and multi volume DTB validations. -->
<ENTITY mastersmil_internal "
<category name='master SMIL document internal validation' context='mastersmil'
&fileIntegrity;
[..]
</category>
" >

<!-- Since it's optional to use a master SMIL file in a DTB, this context has to redefine the failType of the 'exists' test under the mastersmil tests after those have been included. -->
<!ENTITY DTB "
<category name='DTB validation' context='DTB'>
[..]
&mastersmil_internal;
<subcategory name='file validation' context='mastersmil.fi' >
<test context='exists' name='file exists' shortDesc='file not found' longDesc='file not found in DTB directory' failType='warning' failClass='' link='' />
</subcategory>
[..]
</category>
" >

Base elements

The VTM 'validator' element is the base element of the VTM. Nested within the 'validator' element is the 'validatorTestMap' where all categories and tests that are to be used by the validator engine must be declared.

8. The ADTD files

The ADTD files are based upon the W3C DTD specification but overrides and extends it in many ways. There is one ADTD for each XML based document found within a DTB.

File type ADTD file name
NCC ncc.adtd
SMIL smil.adtd
Content document content.adtd
Discinfo document discinfo.adtd
Master SMIL document msmil.adtd
Multi volume NCC multivolume.adtd

The ADTD documents takes three declaration objects; ENTITY, ATTLIST and ATTCORLIST.

The ADTD rules

The ADTD rules system is very similar to the W3C DTD specification. The system is based upon string expressions, occurrence suffixes, separators, entities, character entities, character data and spans.

An ADTD rule set is always enclosed within a pair of parentheses, a typical string expression is enclosed within double (") or single (') quotes.

The ADTD occurrence suffixes Description
+ Must occur one or more times
? Must occur one or zero times
* Must occur zero or more times
{no suffix} Must occur one time

The ADTD separators Description
, Both the expression before and after the ',' must be true and they must follow the given order.
| Either the expression before the '|' OR the expression after it must be true.

This system gives the user a powerful way of controlling the data flow.

Simple ADTD rules example
/* The given data must according to the rules given contain the string 'black' or 'blue' */
('black' | 'blue')

/* The given data must according to the rules given contain 'big' and 'apple' in series. */
('big', 'apple')

/* The given data can according to the rules given contain 'big' but MUST contain 'car', and if 'big' exists, 'car' must be in series. */
(('big')?, 'car')

Declaration of allowed characters are possible to make using the above example, though it get's somewhat ineffective. Using a character data or span is an better option.

A character data set is defined as '[{the characters}]', a span is defined as '[{from character}-{to character}]'. These two methods can also be combined to form the most effective solution.

Extended ADTD rules example
/* The given data must according to the rules given contain one or zero of the characters */
([abcABC])?

/* The given data must according to the rules given contain one or several of the characters explicitly stated or found in between 'a' and 'z' ('a' and 'z' included) */
([0123456789a-z])+
Note that character data sets and spans are case sensitive. Those who are used to the standard DTD set, notice that none of the two NOT signs, '-' and '^', are allowed within an ADTD.

Hexadecimal character entities can be used to replace characters where needed. A hexadecimal character entity is declared trough '#x{hexnumber of character};'.

Entities

Entities are declared using the standard !ENTITY declaration. Entities are parsed immediately, disregarding any standard parsing rules. Recursive parsing is not used, therefore the loading process of a ADTD will fail if an entity is used before it is declared in the file.

No standard identifiers such as CDATA, IDTOKEN etc. are allowed.

Entities are declared trough <!ENTITY % {entityname} "{rules}"> and are referred to by %{entityname}; .

Entity example
<!ENTITY % endsync-val "('last')">

Attribute lists

Attribute lists are declared using the standard !ATTLIST declaration. ATTLIST objects always belongs to a certain element and can contain several attribute declarations for that specific element.

The ATDT ATTLIST declaration differs from the standard W3C DTD specification in the sense that the attribute contents are binding and has a parameter for case sensitivity.

ATTLIST are declared trough <!ATTLIST {element name} {attribute name} {attribute content rules} {case sensitivity (CI for case insensitive, CS for case sensitive)} {DTD legacy (unused, should always be #IMPLIED)} [..] > .

ATTLIST example
<!ATTLIST audio
src %URI-referenceExcerpt; CI #IMPLIED
clip-begin %NPTs; CS #IMPLIED
clip-end %NPTs; CS #IMPLIED
>

The attribute correlation list

The attribute correlation list is a total stepaway from the original W3C DTD specification. The attribute correlation list is declared trough !ATTCORLIST and defines attribute contents rules depending on the contents of other attributes within the same element.

In an attribute correlation list, the first attribute name / rules pair is the controlling one.

Declaration is done trough <!ATTLIST {element} {controlling attributes name} {contents rules} {case sensitivity (CI for case insensitive, CS for case sensitive)} {attribute name} {attribute contents rules} {case sensitivity} [..] # {occurrence suffix} {recommended or required, put 'r' for recommended, leave blank otherwise} >

Attribute correlation list example
/* The following example declares that if a <meta> element is found that has a 'name' attribute that contains 'dc:format' (case sensitive) then the element must have a 'content' attribute with the contents 'Daisy 2.02' (case insensitive). The occurrence suffix is left blank; the combination must exists within the given data once. */

<!ATTCORLIST meta name ('dc:format') CS content ('Daisy 2.02') CI #>

/* The following example declares that if a <meta> element is found that has a 'name' attribute that contains 'dc:date' (case sensitive) then the element must have a 'scheme' attribute with the contents 'YYYY-MM-DD' (case insensitive) and a 'content' attribute that conforms to the form of date. The combination may exists within the given data once but is recommended. */

<!ENTITY % DIGIT "[0-9]">
<!ENTITY % YEAR4 "(%DIGIT;, %DIGIT;, %DIGIT;, %DIGIT;)">
<!ENTITY % MONTH "(('10') | ('11') | ('12') | ('0', ([1-9])))">
<!ENTITY % DAY "(('0', [1-9]) | ([1-2], [0-9]) | ('30' | '31'))">

<!ATTCORLIST meta name ('dc:date') CS scheme ('YYYY-MM-DD') CI contents (%YEAR4;, ('-')?, %MONTH;, ('-')?, %DAY;) CS #?r>
